# Documentation Reality-Check Checklist

## Purpose

This checklist helps the MORE Club maintain honest, transparent documentation that matches our actual organizational capacity and programming. Use this regularly to prevent documentation drift where promises exceed reality.

## Review Schedule

- **Monthly:** Quick scan for obviously outdated information
- **Quarterly:** Full documentation audit using this checklist
- **Before major announcements:** Review all public-facing docs
- **After organizational changes:** Update affected documentation immediately

## General Documentation Principles

Before reviewing specific pages, remind yourself of these principles:

- ☐ **Reality First:** Does this accurately describe what exists NOW?
- ☐ **Tense Check:** Is future/planned language used for things that don't exist yet?
- ☐ **Founding Phase:** Do we still need founding phase notices? Are they accurate?
- ☐ **Response Times:** Can we actually meet the commitments we've stated?
- ☐ **Volunteer Capacity:** Are expectations aligned with volunteer availability?

## Red Flags to Watch For

### Language That Overpromises

- ❌ "We offer..." (when we don't yet)
- ❌ "Members receive..." (when we haven't delivered this)
- ❌ "Weekly rides" or "Every [time period]" (when rides aren't scheduled)
- ❌ "Regular meetings" (when no meeting schedule exists)
- ❌ "Partnership discounts" (when partnerships aren't established)
- ❌ Present tense for future intentions

### Better Alternatives

- ✅ "We're building..."
- ✅ "We plan to offer..."
- ✅ "Founding members will help establish..."
- ✅ "Once launched, we'll..."
- ✅ "We're working toward..."
- ✅ Clear distinction between current and planned

## Page-by-Page Checklist

### Homepage / README.md

**Reality Checks:**
- ☐ Status section exists and is current
- ☐ "Live Now" items are actually available
- ☐ "In Development" items are actually being developed
- ☐ Contact information works (test the email)
- ☐ Links to other pages work
- ☐ Founding phase status is accurate

**Red Flags:**
- ☐ Claims about active programming that doesn't exist
- ☐ Member counts that aren't verified
- ☐ Features listed as live that aren't

**Action if Issues Found:**
- Update status section
- Move items between "Live" and "In Development" as needed
- Add realistic timelines if appropriate

---

### About Page

**Reality Checks:**
- ☐ Mission statement still accurate
- ☐ "What We're Building" matches actual plans
- ☐ Values section reflects actual community culture
- ☐ Founding member opportunities are genuine
- ☐ Call-to-action is appropriate for current phase

**Red Flags:**
- ☐ Present tense for things that don't exist
- ☐ Membership benefits listed that aren't available
- ☐ Team/leadership structure that doesn't exist

**Action if Issues Found:**
- Update tense (present vs. future/planning)
- Revise benefits to match reality or mark as planned
- Update founding phase notice if transitioning

---

### Events Page

**Reality Checks:**
- ☐ Are there ACTUAL events scheduled? (with real dates)
- ☐ If no events scheduled, does page say so honestly?
- ☐ Event frequency claims match reality ("weekly" vs "planned weekly")
- ☐ Can the organization actually deliver listed events?
- ☐ Registration process (if mentioned) actually works

**Red Flags:**
- ☐ "TBD" dates that have been TBD for months
- ☐ Regular schedule claimed without actual scheduled events
- ☐ Event types listed that no one is organizing
- ☐ Past events not marked as past

**Action if Issues Found:**
- Remove or update "TBD" placeholders
- Change to "planned programming" if no actual schedule
- Archive past events or create "past events" section
- Add honest timeline ("We're working toward launching rides in [season]")

---

### Contact Page

**Reality Checks:**
- ☐ Email address is monitored
- ☐ Response time promise is realistic (test it!)
- ☐ Meeting information is current or marked as TBD with context
- ☐ Social media links work and are active
- ☐ Contact form (if exists) actually works

**Red Flags:**
- ☐ Response times that can't be met
- ☐ Meeting schedule that doesn't exist
- ☐ Multiple contact methods that aren't actually monitored
- ☐ Office hours or availability that isn't realistic

**Action if Issues Found:**
- Test email response time over 2 weeks, adjust promise
- Update meeting info to "establishing schedule based on member input"
- Remove or mark as "planned" any contact methods not yet available

---

### Rides/Trail Information Pages

**Reality Checks:**
- ☐ Actual scheduled rides have real dates
- ☐ Trail information is current and verified
- ☐ Difficulty ratings are honest and consistent
- ☐ "Featured rides" or examples are labeled as such if not scheduled
- ☐ Ride reports are from actual rides that happened

**Red Flags:**
- ☐ "Upcoming rides" with no actual dates
- ☐ Trail conditions that haven't been verified recently
- ☐ Claims about guided rides with no guides
- ☐ Placeholder content presented as real

**Action if Issues Found:**
- Label examples as "planned rides" or "ride concepts"
- Add dates to trail info ("last verified: [date]")
- Create separate sections for "scheduled" vs "planned" rides
- Archive old ride reports with dates

---

### Membership Page

**Reality Checks:**
- ☐ Membership benefits match what's actually provided
- ☐ If no formal membership exists, does page say so?
- ☐ Benefits requiring partnerships are actually available
- ☐ Member resources listed are accessible
- ☐ Membership process (if described) works

**Red Flags:**
- ☐ Benefits listed that don't exist
- ☐ "Member-only" resources that aren't built yet
- ☐ Partnership discounts without actual partnerships
- ☐ Membership fees mentioned without structure in place

**Action if Issues Found:**
- Change "benefits" to "founding member opportunities"
- Separate "current" from "planned" benefits
- Remove or mark as coming soon: unavailable resources
- Add transparency about membership structure development

---

### Discord/Community Pages

**Reality Checks:**
- ☐ Discord server actually exists and invite works
- ☐ Channels described actually exist in the server
- ☐ Activity level accurately represented
- ☐ Moderation/leadership exists as described
- ☐ Community guidelines are enforced

**Red Flags:**
- ☐ Describing vibrant community that doesn't exist
- ☐ Channels listed that aren't created
- ☐ Activity promises that aren't met
- ☐ Support availability that isn't realistic

**Action if Issues Found:**
- Test Discord invite link
- Update channel list to match actual server
- Adjust language about community size/activity
- Note that community is growing/building

## Specific Section Checks

### Safety Documentation

**Reality Checks:**
- ☐ Information is accurate and current
- ☐ Emergency contacts work
- ☐ Procedures are tested or vetted by experts
- ☐ Disclaimers are appropriate
- ☐ Sources are cited where needed

**Action:** Safety documentation should be reviewed by qualified person quarterly.

---

### Governance Documents

**Reality Checks:**
- ☐ Decision-making process matches actual practice
- ☐ Roles described exist and are filled
- ☐ Founding phase status is accurate
- ☐ Last updated date is current

**Action:** Update governance docs when organizational structure changes.

## Transition Checks: Moving from Founding Phase

When considering removing "founding phase" language, verify:

- ☐ Regular event schedule established (minimum 3 months consistent programming)
- ☐ Core group of 10+ active members
- ☐ Membership structure decided and implemented
- ☐ Regular communication cadence established
- ☐ At least one established partnership or benefit
- ☐ Governance structure functioning as described
- ☐ Financial model (if any) is working
- ☐ Leadership roles filled and functioning

**Don't rush this transition!** Being in "founding phase" is honest and attractive to the right people.

## Quarterly Audit Process

### Step 1: Gather Information (Week 1)

- ☐ Review what events/rides actually happened in last quarter
- ☐ Count active community members/participants
- ☐ List new features/resources that became available
- ☐ Check what "in development" items actually progressed
- ☐ Test response times and contact methods
- ☐ Review Discord/community activity levels

### Step 2: Review Documentation (Week 2)

- ☐ Go through page-by-page checklist above
- ☐ Note discrepancies between docs and reality
- ☐ Identify outdated information
- ☐ Check all links and contact methods
- ☐ Review for tone and language issues

### Step 3: Update Documentation (Week 3)

- ☐ Make necessary corrections
- ☐ Update status sections
- ☐ Adjust promises/commitments to match capacity
- ☐ Add dates to time-sensitive information
- ☐ Update "last reviewed" dates in key documents

### Step 4: Communicate Changes (Week 4)

- ☐ Notify community of any significant changes
- ☐ Update GitHub repository
- ☐ Post updates to Discord
- ☐ Send email if major changes affect members

## Emergency Checks

Run this quick check if you get negative feedback about documentation:

1. ☐ What specific claim was made in documentation?
2. ☐ Is that claim true? Can we back it up?
3. ☐ If not true, when did it become untrue?
4. ☐ Update documentation immediately
5. ☐ Acknowledge issue transparently to community
6. ☐ Review similar claims elsewhere in documentation

## Questions to Ask Yourself

Before publishing any new documentation or updates:

1. **Is it true?** - Does this accurately describe current reality?
2. **Is it complete?** - Have I disclosed important limitations or development status?
3. **Is it fair?** - Does this set appropriate expectations?
4. **Is it sustainable?** - Can we actually deliver on these commitments?
5. **Is it clear?** - Will readers understand what's available now vs. planned?

If you can't answer "yes" to all five, revise before publishing.

## Success Metrics

Healthy documentation should result in:

- ✅ New members have accurate expectations
- ✅ Community feedback is positive about transparency
- ✅ Actual participation matches what documentation suggests
- ✅ Few surprises or disappointments about availability
- ✅ Growing trust in the organization

If you're seeing:
- ❌ New members disappointed by what they find
- ❌ Questions about "where are the rides/events mentioned?"
- ❌ Confusion about organizational status
- ❌ Feedback about overpromising

Then documentation needs immediate review and update.

## Remember

**Good documentation is:**
- ✅ Honest about current state
- ✅ Clear about what's planned vs. available
- ✅ Enthusiastic without overpromising
- ✅ Updated regularly
- ✅ Aligned with actual capacity

**Bad documentation:**
- ❌ Creates false expectations
- ❌ Damages trust when reality doesn't match
- ❌ Leads to disappointed community members
- ❌ Harms reputation long-term

**When in doubt:** Be more conservative in claims, more transparent about status, more realistic about timelines.
